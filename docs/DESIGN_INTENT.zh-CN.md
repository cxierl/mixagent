# MixAgent 设计意图

## 产品定位

MixAgent 不是一个“多几个聊天窗口”的 AI 工具，而是一个面向真实生产场景的多智能体协作工作台。

它的设计目标很明确：

- 让一个用户可以像带一个小型 AI 团队一样工作
- 让团队、群聊、策略、产物都可以复用
- 让讨论过程不只是停留在聊天，而是沉淀为可追踪、可导出、可交付的成果

## 为什么这样设计

### 1. 以房间为工作单元，而不是一次性对话

真实工作通常不是一次提问、一次回答就结束。

所以 MixAgent 采用“群聊房间”作为主容器。房间里可以承载：

- 固定团队成员
- 讨论历史
- 当前目标
- 当前文件
- 当前成果

这样房间本身就变成一个可持续使用的工作空间。

### 2. 以团队复用为前提，而不是每次重建角色

你已经明确提出：同一个团队要能应对不同业务需求。

因此 MixAgent 采用“团队长期存在、产物模式按次选择”的策略：

- 团队是长期结构
- 群聊是具体工作空间
- 产物模式是本次命令的上下文

这避免了“为了不同结果形式反复重建群聊”的低效操作。

### 3. 以系统记录代替记录员席位

记录员作为显式智能体有三个问题：

- 占一个发言位
- 打断真实讨论节奏
- 记录质量受模型输出波动影响

因此 MixAgent 改为系统层留痕：

- 保存原始消息
- 提炼结构化状态
- 跟踪成果索引
- 为后续导出和续写提供支撑

记录成为系统能力，而不是聊天角色。

### 4. 以“一个可见助手 + 一个隐藏控制层”降低复杂度

用户只需要看到一个默认入口：`智能助手`。

但系统内部仍然拆分为两层：

- 可见助手：负责和用户沟通、提案、确认
- 隐藏控制层：负责真正执行配置、创建、保存、生成、更新

这样既不牺牲能力，也不会把系统复杂度直接压到界面上。

## 底层逻辑

## 1. 配置持久化

MixAgent 的核心设计原则之一是：更新不能冲掉用户配置。

因此模型、技能、智能体、团队、群聊、发言策略都做了本地持久化。

当前关键存储包括：

- `ms_v5`：模型实例
- `mix_model_services_v1`：隐藏的共享服务商层
- `csk_v5`：自定义技能
- `mix_agents_v1`：智能体
- `mix_projects_v1`：群聊/项目
- `mix_teams_v1`：团队
- `mix_team_members_v1`：团队成员关系
- `mix_workflow_cfg_v2`：发言策略

## 2. 模型层与服务商层分离

很多模型会共用同一个：

- Base URL
- 格式
- API Key

所以底层做了“服务商层”抽象，负责共享连接参数。

但前台 UI 不再直接把这层暴露出来，而是回归为：

- 左侧模型列表
- 右侧模型配置

这是为了让用户只面对当前任务，不面对底层实现结构。

## 3. 运行单元（Run）思路

每次命令不只是聊天，而是一次“运行单元”。

一个 Run 会记录：

- 本次命令
- 当前产物模式
- 讨论过程
- 最终成果
- 元数据与索引

这样同一个房间里不同任务不会混成一团。

## 4. 产物模式按次选择

MixAgent 当前支持：

- 快速文本
- 单文件
- 多文件
- 工程文件

这样既能支持一句结论，也能支持报告包、设定包、长篇工程。

## 一键安装与自动更新

### 一键安装

MixAgent 通过 NSIS 打包为 Windows 安装包，支持用户直接双击安装。

### 自动更新

从 `v0.1.2` 开始，桌面端已支持：

- 启动自动检查更新
- 窗口重新激活时按节流策略复查
- 有新版本时弹出提示
- 下载时显示进度条
- 下载后支持一键重启安装

前提是发布时把以下文件上传到 GitHub Releases：

- 安装包 `.exe`
- `.blockmap`
- `latest.yml`

## 适合的使用场景

MixAgent 适合这类任务：

- 小说世界观与人物团队协作
- 方案设计与论证
- 产品定义与执行计划
- 长篇内容生产
- 多角色审查与风险把关

它不追求“一个模型回答一切”，而追求“让多个角色在可控结构里协作，并沉淀出真实成果”。
